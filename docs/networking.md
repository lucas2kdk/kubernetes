# Networking and network security

How traffic is governed inside the cluster: the CNI, the cluster-wide
default-deny model, the individual Cilium policies, the four-list lockstep that
keeps platform namespaces exempt, and how to add a platform namespace correctly.

See also: [architecture](architecture.md) · [operations](operations.md) ·
[Kyverno policies](policies.md) · [top-level README](../README.md).

## Cilium as the CNI

The cluster runs [Cilium](https://cilium.io/) as its CNI. **Cilium itself is
managed out-of-band by Terraform, not by this repository** — Terraform installs
the Cilium agent/operator and, critically, the `cilium.io` CRDs
(`CiliumClusterwideNetworkPolicy`, `CiliumNetworkPolicy`). This repo only ships
*policy objects* that those CRDs back; the `network-policies` fleet
`Kustomization` is a root with no `dependsOn`, but it implicitly requires the
Cilium CRDs to already exist (hence its inability to self-bootstrap the CNI).

> **Hubble.** Cilium's Hubble observability layer (flow visibility, the
> service map, `hubble observe`) is likewise provisioned out-of-band via
> Terraform. It is not configured by anything in this repo.

Because policy enforcement is delegated to Cilium, the network guardrails are
expressed as Cilium CRDs (`platform/base/network/`) rather than upstream
`NetworkPolicy`, and the per-tenant baseline policy generated by Kyverno is a
`CiliumNetworkPolicy` (see [Kyverno policies](policies.md)).

## The cluster-wide default-deny model

The foundation is a **cluster-wide default-deny** applied to *workload*
namespaces only. Platform and system namespaces are excluded so the control
plane, GitOps reconciliation and admission webhooks keep working.

The key Cilium semantic that shapes every file here: **any rule that selects an
endpoint flips that endpoint into default-deny for that direction.** So the
policies cannot use an empty `endpointSelector: {}` (that would select every pod
in the cluster, including `kube-system`, `flux-system` and `kyverno`, and put
them into deny with no matching allow — breaking CoreDNS, the Kyverno webhook
and Flux's egress to GitHub). Instead each policy selects "every namespace
**except** the platform ones" via a `NotIn` match expression on
`k8s:io.kubernetes.pod.namespace`.

Under default-deny, a selected workload pod may only:

| Direction | Allowed |
|-----------|---------|
| Ingress | `host`, `remote-node`, `kube-apiserver` entities (kubelet health/liveness probes, API server) |
| Egress | `host`, `remote-node`, `kube-apiserver` entities; DNS (UDP/TCP `53`) to CoreDNS (`kube-system` / `k8s-app=kube-dns`) |

**Everything else needs an explicit allow.** Same-namespace pod-to-pod traffic
is restored per-namespace by the Kyverno-generated baseline `CiliumNetworkPolicy`
(see [policies](policies.md)); cross-cutting grants live in the `allow-*` files
below.

## The policy files (`platform/base/network/`)

All four are reconciled by the `network` base's
[`kustomization.yaml`](../platform/base/network/kustomization.yaml).

| File | Kind | Effect |
|------|------|--------|
| [`default-deny.yaml`](../platform/base/network/default-deny.yaml) | `CiliumClusterwideNetworkPolicy` | The foundation. Selects all workload namespaces (`NotIn` the platform list) and grants only host/`remote-node`/`kube-apiserver` ingress+egress and DNS egress to CoreDNS. All other traffic is denied. |
| [`deny-cloud-metadata.yaml`](../platform/base/network/deny-cloud-metadata.yaml) | `CiliumClusterwideNetworkPolicy` | Anti-SSRF defense in depth. An `egressDeny` to the link-local cloud/instance-metadata IP `169.254.169.254/32`, so a compromised workload cannot harvest node/cloud credentials from the metadata endpoint. Scoped with the **same** `NotIn` exclusion list as default-deny — deliberately *not* `endpointSelector: {}`, which would flip every pod (kube-system, flux-system, kyverno) into egress-deny and break DNS/webhooks/Flux cluster-wide. |
| [`allow-ingress-from-traefik.yaml`](../platform/base/network/allow-ingress-from-traefik.yaml) | `CiliumClusterwideNetworkPolicy` | Additive allow: lets workload pods receive ingress from the Traefik ingress controller (`fromEndpoints` matching namespace `traefik`). Without it the default-deny blocks all traffic to pods behind the ingress. Pairs with the per-namespace baseline CNP. |
| [`allow-ingress-from-monitoring.yaml`](../platform/base/network/allow-ingress-from-monitoring.yaml) | `CiliumClusterwideNetworkPolicy` | Additive allow: lets workload pods receive ingress from the Prometheus stack in the `monitoring` namespace so scrape jobs reach their targets. Mirrors the traefik policy's endpoint selector. |

## The four-list lockstep

Four hand-maintained lists must stay **set-identical**. Three are the Cilium
`NotIn values:` lists; the fourth is the Kyverno member.

| # | File | Field |
|---|------|-------|
| 1 | `platform/base/network/default-deny.yaml` | `spec.endpointSelector.matchExpressions[NotIn].values` |
| 2 | `platform/base/network/deny-cloud-metadata.yaml` | `spec.endpointSelector.matchExpressions[NotIn].values` |
| 3 | `platform/base/network/allow-ingress-from-traefik.yaml` | `spec.endpointSelector.matchExpressions[NotIn].values` |
| 4 | `platform/base/policies/generate-baseline-netpol.yaml` | `spec.rules[].exclude.any[].resources.names` |

These are the namespaces **exempt** from the tenant guardrails. The canonical
set today:

```
kube-system, kube-node-lease, kube-public, flux-system,
kyverno, policy-reporter, cert-manager, external-secrets,
tailscale, traefik, monitoring, headlamp, tsidp,
trivy-system, longhorn-system
```

### Why they can't be deduplicated

They live in **two different kustomize builds** with different field semantics
(Cilium `NotIn` string values vs. a Kyverno `exclude...names` list), so there is
no single object to share. The obvious "one source via Flux variable
substitution" approach is also out: `kubeconform` schema-validates the committed
Cilium CRD manifests, and a Flux-substituted YAML *list* would leave a `${var}`
**string** sitting where the CRD schema demands an **array** — validation fails
before the substitution ever happens at runtime. So the lists stay as literal
strings, edited by hand, in lockstep.

### The drift guard (`just namespaces`)

Because the lists are hand-maintained, drift is caught by
[`scripts/check-namespace-lists.sh`](../scripts/check-namespace-lists.sh), run as
`just namespaces` (part of `just check` and the PR gate). It extracts the
namespace set from each of the four files (via `yq`, falling back to
`python3`+PyYAML), sorts them, and **fails the build if any two diverge** —
printing exactly which namespace is missing from / extra in which file. You can
no longer merge a PR where the four lists disagree. The script compares only; it
changes no policy behaviour.

### The 2026-06-12 longhorn-system CSI outage

The guard exists because of a real outage. `longhorn-system` (Terraform-managed
storage) is a **special case**: it must appear in **all four** lists, because it
deliberately has *no* baseline CNP and relies on being outside every policy to
keep CSI ↔ kube-apiserver traffic flowing.

On **2026-06-12** the lists drifted: `longhorn-system` was present in three but
the set fell out of lockstep, and `generate-baseline-netpol` injected a baseline
CNP into `longhorn-system`. Selecting its pods flipped them into default-deny,
which (with no allow rule for the `kube-apiserver` entity) cut the CSI sidecars
off from the API server — the Longhorn provisioner crash-looped, PVCs stuck
`Pending`, and the cluster suffered a storage outage. The lockstep guard now
makes that exact class of drift impossible to merge. (The deny-cloud-metadata
and allow-ingress-from-traefik entries for `longhorn-system` carry the same
rationale in inline comments: any rule selecting its pods enables enforcement on
that direction and severs traffic the storage layer needs.)

### `allow-ingress-from-monitoring` is NOT in the guarded set

This is correct and intentional. `allow-ingress-from-monitoring.yaml` mirrors
the same `NotIn` exclusion list, **but it is not one of the four files the drift
guard checks** — verified against `scripts/check-namespace-lists.sh`, whose
`cilium_files` array lists only `default-deny.yaml`, `deny-cloud-metadata.yaml`
and `allow-ingress-from-traefik.yaml` (plus the Kyverno file). Its own header
calls this out and asks you to keep it aligned by hand.

Note also that its list is *not even identical* to the guarded set: it
additionally contains the scaffolded-stub namespaces **`netdata`** and
**`vector`** (which have component bases but no fleet `Kustomization` yet).
**If you add a platform namespace, you must edit this file too, by hand** — the
guard will not remind you — or Prometheus loses scrape access to the new
namespace's workloads.

## How to add a platform namespace correctly

A platform namespace must be exempted from the guardrails in **three** places
(see the README's [Platform namespace convention](../README.md#platform-namespace-convention)).
Only the first is automatic:

1. **Kyverno pod/Flux policies** — *automatic*. The Pod / Kustomization /
   HelmRelease policies exclude via
   `namespaceSelector: matchLabels: platform.io/managed: "true"`, so putting
   that label on the component's `platform/base/<component>/namespace.yaml` is
   enough. **Exception:** `generate-baseline-netpol.yaml` matches on the
   `Namespace` *kind* and keeps an explicit `names:` list — add the new
   namespace there by hand (it is list #4 above).

2. **Cilium cluster-wide policies** — *manual*. Add the namespace to all three
   `NotIn` lists (`default-deny`, `deny-cloud-metadata`,
   `allow-ingress-from-traefik`). Forgetting this default-denies the new
   namespace and breaks its ingress/egress. Cilium can't match arbitrary
   namespace labels in a `NotIn`, which is why these are literal strings rather
   than a `platform.io/managed` selector.

3. **The four system namespaces** (`kube-system`, `kube-node-lease`,
   `kube-public`, `flux-system`) have no `namespace.yaml` in this repo to label,
   so they stay as explicit entries in the Kyverno static lists too.

In practice: edit lists #1–#4 in lockstep (the `just namespaces` guard enforces
this), **and** remember to also edit the un-guarded
`allow-ingress-from-monitoring.yaml`. Then run `just namespaces` (or `just
check`) locally before pushing.
